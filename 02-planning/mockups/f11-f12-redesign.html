Co dochodzi vs. dziś

  • Popup overlay — zero pseudo-sesji w list-sessions.
  • Live metryki: uptime, last cmd, CPU%, Mem.
  • Kill z potwierdzeniem (default: cancel).
  • New auto-przydziela najniższy wolny F-slot.
  • Slot vs. session — wolne F6/F7/F9/F10 widoczne jako placeholder.
  • Sleep state — kompatybilny z auto-suspend.
  • Numeric jump — 5 stawia kursor na F5.

Co tu się dzieje

  • Auto-generowane z tmux list-keys -T root + adnotacje # @group: navigation @desc: … w tmux.conf.
  • Live — odzwierciedla bieżący config, nie stary docstring.
  • Searchable — / filtruje fuzzy.
  • Groupable — g przełącza grupowanie (kategoria / alfa / by-key).
  • Toggle — F12 otwiera i zamyka tym samym klawiszem.
  • Brak sleep infinity, brak pseudo-sesji.

CEO ReviewPRODUCT

VERDICT: GO (with conditions)

Why this matters

pTTY's pitch is „SSH + reboot? Twoja sesja żyje". F11 i F12 to jedyne interaktywne UI projektu. Dziś F11 jest dwustopniowe, F12 jest statycznym tekstem, oba są zepsute (Task 013 — paths). Pierwsze wrażenie z release'u v0.2 zależy od tych dwóch ekranów. Reszta projektu to plumbing.

Strategiczna ocena

Time-to-„wow"F11-popup z metrykami daje natychmiastowe poczucie kontroli. Demo-friendly. Tweet-friendly.
Differentiationvs. gołe tmux: branding (pTTY), uptime/CPU per-console, slot-aware. vs. Zellij: lżejsze, bash-only, instaluje się w 30s.
Story do READMEAnimowany GIF F11 robi robotę. F12 which-key — drugi GIF. Bez tego release ma trzy strony tekstu i nic do pokazania.
RiskSingle-key UX odkrywany na bieżąco; trzeba dobry on-boarding (pierwsze otwarcie F11 → tooltip „press ? for help").

Conditions

  1. Task 013 ship-blocker — bez tego v0.2 nie ma sensu, bo nikt nie zobaczy F11/F12.
  2. Asset budget — 2 GIF-y do README (F11 demo, F12 demo) przed merge'em do master.
  3. Decision: sessionx adopt vs. own — nie zwlekać, decyzja w tym sprincie. Każdy tydzień własnej implementacji to tydzień nieukończonego release'u.
  4. Naming — jeśli rename (Task 012) idzie razem z v0.2, F11/F12 muszą pokazywać „pTTY" w tytule popup. Spójność marki w pierwszym kontakcie.

Co odpuszczam (na teraz)

  • Drag-to-reorder F-slots — v0.3.

Engineering ReviewTECH

VERDICT: CONDITIONAL GO

Architecture call-outs

Path universality set-environment -g PTTY_DIR + sed-template w install.sh to standard. OK. Egzekwować grep -r "vps/sessions" w pre-commit / CI — żaden hardcode się nie wśliźnie.
Popup compat tmux ≥ 3.2 wymagane. Repo dziś ma fallback (display-popup -h 1 'echo test') do split-window — zachować dla starych distro (Debian 11 ma 3.1!). F11/F12 w tym samym wzorcu co Ctrl+H.
Fzf vs. gum vs. own Bash + fzf --bind obsłuży 100% F11. Gum dla F12 (lepsze grupowanie). Nie wprowadzać Go w v0.2 (Task 017 = v1.0).
Stale parsing Dziś awk '{print $3}' z display rows = bomba. Przepisać: każdy wiersz to $F_KEY\t$STATUS\t$SESSION\t$JSON, fzf renderuje przez format spec, akcje czytają tab-separated → solidnie.
Help auto-gen Annotation w tmux.conf: # @group:nav @desc:Jump to console 1. Parser w bash, 30 linii. Jeden punkt prawdy = klucz w conf, nie duplikowany w docstr.
Tests bats (już zaplanowane w Task 004). Smoke: tmux -f tmux.conf send-keys C-F11, tmux capture-pane, assert że nagłówek pTTY Manager się pojawia. Nie testować pikseli, testować tekst.

Risks & mitigations

  • tmux 3.0/3.1 users (Debian 11, RHEL 8) — popup nie istnieje. Mitigation: detekcja jak Ctrl+H, fallback do split-window. Nie blokować popup-only release'u na starych systemach; dokumentacja minimalnej wersji.
  • fzf version skew — stare fzf nie ma --bind reload. install.sh sprawdza fzf --version >= 0.30, podpowiada upgrade.
  • Race podczas restart — obecny mission-control.sh używa lock-file ($temp_dir/restart-${session}.lock). Zachować, dorobić timeout+cleanup w trapach.
  • Single-key destructive (K) — zawsze confirm dialog, default: Cancel (kursor na „No"). : jako command-prompt na destrukty długim hasłem? overkill na razie.
