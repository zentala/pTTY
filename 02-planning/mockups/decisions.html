Q1Wzorzec uniwersalnych ścieżek

Jak tmux.conf ma znaleźć skrypty pTTY niezależnie od miejsca instalacji?

A. Env-var PTTY_DIR + sed-template REKOMENDACJA

Pierwsza linia conf: set-environment -g PTTY_DIR "__PTTY_DIR__". install.sh robi sed przy instalacji, podmienia placeholder. Bindingi: #{PTTY_DIR}/src/foo.sh.

Plusy

  • Natywny tmux mechanizm, format-spec działa w run-shell i display-popup
  • Jeden punkt prawdy — zmienna w sesji, nie ścieżka rozsiana po 20 plikach
  • Override przez env: PTTY_DIR=$PWD ./install.sh dla dev
  • Czytelne w debug: tmux show-environment -g PTTY_DIR

Minusy

  • Wymaga kroku sed w instalatorze (drobna złożoność)
  • tmux.conf w repo zawiera placeholder — nie odpalisz „raw" bez install

B. Symlink do ~/.ptty OK

install.sh tworzy ln -s $PWD ~/.ptty. Wszystkie ścieżki w conf hardcoded na ~/.ptty/....

Plusy

  • Najprostszy do napisania (1 linia w installerze)
  • tmux.conf w repo działa „raw" jeśli zrobisz sym­link ręcznie

Minusy

  • Narzuca konwencję — user nie może zainstalować w /opt/ptty bez kombinacji
  • Symlinki w $HOME bywają problemem przy backupach / sync (Dropbox, Syncthing)
  • Łamie się przy wielu instancjach (test + prod na tym samym koncie)

C. Config file ~/.config/ptty/dir SKIP

Każdy skrypt zaczyna od PTTY_DIR=$(cat ~/.config/ptty/dir).

Plusy

  • Najbardziej elastyczny — wiele instancji, runtime override

Minusy

  • Ekstra plik, ekstra I/O w każdym skrypcie
  • tmux.conf nadal nie wie o tej ścieżce — trzeba i tak template'ować
  • Over-engineering jak na ten problem

Moja rekomendacja

A — env-var + sed-template. Jednorazowy koszt w installerze, czyste runtime, zero magii w skryptach. To wzorzec używany przez TPM, oh-my-zsh, neovim plugins.

Q2Plugin czy własna implementacja

F11 (Manager) i F12 (Help) — adoptować istniejące pluginy tmux czy budować od zera?

A. Adopt: sessionx + which-key FAST

omerxx/tmux-sessionx dla F11, alberti42/tmux-which-key dla F12. Instalujemy przez TPM w install.sh.

Plusy

  • Time-to-ship: 2 dni vs. 7 dni
  • Maintained externally, my dostajemy bug-fixy za darmo
  • Sprawdzone na tysiącach setupów — corner cases już znalezione

Minusy

  • Sessionx nie ma kolumn z metrykami (CPU, Mem, last cmd) — branding/UX podstawowy
  • Sessionx pokazuje wszystkie sesje tmux, nie zna konceptu „F-slot 1..10" pTTY
  • Which-key wymaga osobnego pliku z keymapą — duplikuje tmux.conf
  • Dwie kolejne zależności w install.sh, więcej rzeczy może paść
  • pTTY traci tożsamość — wygląda jak „tmux z pluginami", a ma być produktem

B. Hybryda: własny F11, plugin F12 REKOMENDACJA

F11 (Manager) własny — core differentiator pTTY (F-sloty, metryki, sleep state). F12 (Help) na tmux-which-key — pomocny, ale nie buduje marki.

Plusy

  • F11 = nasz „wow", pełna kontrola UX, GIF do README
  • F12 = solved problem, nie wymyślamy koła
  • Time-to-ship realistyczny: ~5 dni
  • Mniejsza powierzchnia własnego kodu = mniejszy maintenance

Minusy

  • Dwa style estetyki w popupach (own vs. plugin) — trzeba pilnować spójności
  • Eval which-key trzeba zrobić zanim się commit­niesz (Task 016)

C. Wszystko własne PURE

F11 i F12 napisane w pTTY, zero pluginów tmux poza TPM (dla resurrect+continuum).

Plusy

  • Maksymalna kontrola, zero kompromisów estetycznych
  • Mniej zależności do install
  • Najczystsza tożsamość produktu

Minusy

  • +5 dni pracy na F12 którego nikt poza pTTY nie odróżni od which-key
  • My utrzymujemy parser list-keys, fuzzy filter, grupowanie — wszystko
  • Risk: F12 ląduje w „good enough", podczas gdy plugin jest „polished"

Moja rekomendacja

B — hybryda. Zasada: own to co buduje markę i daje USP (F11 z F-slotami i metrykami), adopt to co jest commodity (F12 = lista keybindów). Zachowujemy resurrect+continuum jako adopt (Task 016), bo „persistent" branding tego wymaga.

Q3Klawisz „kill" w F11

vi-style j/k nawigacja koliduje z naturalnym k = kill. Jak to rozwiązać?

A. K (shift) REKOMENDACJA

Mała k = up (vi), duża K = kill. Standard w lazygit, k9s, btop.

Plusy

  • Konwencja TUI — userzy lazygit/k9s znają wzorzec
  • Shift = „intent" = naturalny gating dla destrukcji
  • Mnemonic: K**ill

Minusy

  • Trzeba pamiętać o shift — sub-second slower vs. plain key

B. x = kill OK

Wzorzec z vim (x = delete char), gh CLI, niektórych file managerów.

Plusy

  • Bez shift, szybsze
  • Mnemonic: xterminate

Minusy

  • Niespójne z lazygit/k9s ekosystemem
  • Pojedynczy klawisz na destrukcję = większe ryzyko mistype (mimo confirm)

C. d = kill (delete) SKIP

Wzorzec mutt, ranger, niektórych mail clientów.

Plusy

  • Mnemonic: delete

Minusy

  • Konflikt z Ctrl+D (safe detach) — confusing
  • vim: dd = cut line — nie kill, ale podobnie destrukcyjne, miesza modele
  • „delete" sugeruje że można undo — kill sesji jest final

Moja rekomendacja

A — K (shift). Spójność z lazygit/k9s, naturalny gating przez shift, mnemonic. Zostawiamy k = up dla vi-people, którzy są dominującą populacją w terminalu.

Q4Wsparcie dla tmux < 3.2 (Debian 11, RHEL 8)

display-popup dodano w tmux 3.2. Debian 11 ma 3.1, RHEL 8 ma 2.7. Czy wspieramy fallback?

A. Popup-only, dokumentacja minimalnej wersji REKOMENDACJA

README mówi „tmux ≥ 3.2 wymagane". install.sh sprawdza wersję, blokuje instalację z czytelnym komunikatem + linkiem do upgrade'u.

Plusy

  • Jeden code path, czysty kod, łatwo testować
  • tmux 3.2 wyszedł 2021-02 — ma 5 lat, każde aktualne distro ma nowszy
  • Debian 11 (Bullseye) ma EOL czerwiec 2026 — irrelewantne dla v0.2 release'u
  • Backport tmux jest trywialny (apt install tmux z bullseye-backports)

Minusy

  • Część korpo userów na RHEL 8 / Amazon Linux 2 nie zainstaluje bez sudo

B. Dual code path: popup + split-window fallback SAFE

Detekcja jak w obecnym Ctrl+H: display-popup -h 1 'echo test', jeśli failuje → split-window. Każda funkcja ma dwie ścieżki.

Plusy

  • Działa wszędzie, włącznie z legacy
  • Wzorzec już użyty w repo (Ctrl+H, Ctrl+R)

Minusy

  • 2× tyle kodu, 2× tyle testów, 2× tyle bugów
  • UX w split-window jest znacząco gorszy — popup overlay vs. dzielenie pane'a
  • Część funkcji popupa (filter, scrollback) nie ma sensu w split — degradacja niespójna
  • Long-term ciężar: każdy nowy feature ma dwie implementacje

Moja rekomendacja

A — popup-only. tmux 3.2 jest stary, fallback to permanentny długo po tym jak Debian 11 umrze. Lepszy build kontaktu („wymaga tmux ≥ 3.2 — oto jak zainstalować") niż wieczny dwu-tor.

Ale: w install.sh daj copy-paste'owalne komendy upgrade'u dla popularnych distro: Debian/Ubuntu (backports), RHEL (EPEL/source), macOS (brew). To 20 linii bash, zamyka temat.

Q5Tytuł popupa & branding

Co pokazujemy w nagłówku F11/F12 i innych powierzchniach?

A. „pTTY" — pełny rebrand z v0.2 REKOMENDACJA

Popup tytuł: pTTY Manager, pTTY Help. README, install, GitHub repo, GitHub Pages — wszystko jako pTTY (Task 012).

Plusy

  • Jeden release = jedna spójna marka, brak okresu „też się nazywa…"
  • v0.2 to pierwsza publiczna wersja — najlepszy moment
  • Krótko, brand-friendly, domena ptty.zentala.io już zaplanowana
  • Technicznie precyzyjne (persistent pseudo-TTY)

Minusy

  • Trzeba zrobić rename razem (Task 012 + 013 + 014 + 015 = jeden release)
  • GitHub redirect zadziała ze starym URL, ale stare gwiazdki/issues są pod starą nazwą

B. „Persistent Console" zachowawczo SAFE

Popup: Persistent Console — Manager. Rename na pTTY w v0.3.

Plusy

  • Mniej zmian w v0.2, mniej ryzyka regresji
  • Można zobaczyć reakcję community na obecną nazwę przed rebrandingiem

Minusy

  • Dwie zmiany marki w 2 release'ach = dwa razy zamieszanie
  • „Persistent Console" jest długie, w popupie wygląda ciężko
  • v0.2 launch będzie pod „starą" nazwą, momentum przepada przy v0.3 rename

Moja rekomendacja

A — pełny rebrand z v0.2. Pierwsza publiczna wersja to najlepszy i jedyny tani moment na rename. Każdy późniejszy rebranding kosztuje 10× tyle (issues, blog posts, linki, pamięć użytkowników). Task 012 + 013 + 014 + 015 jako jeden „v0.2 = pTTY launch".
